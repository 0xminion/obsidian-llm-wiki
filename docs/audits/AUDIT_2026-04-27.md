```
Telemetry.jsonl
       │
       ▼
┌─────────────┐     ┌─────────────┐
│  Compile    │     │  Lint       │
│  (merge,    │     │  (15 checks)│
│   crosslink)│     │             │
└─────────────┘     └─────────────┘
```

**Pipeline modes:**
- **Template mode** (default): Deterministic Python templating + small LLM call for insights
- **Agent mode** (`--agent`): Full Hermes subprocess per batch (slower, 600s timeout)
- **Review mode** (`--review`): Stage files for approval before writing
- **Resume** (`--resume`): Skip Stages 1+2, re-run Stage 3

## New Findings

### MEDIUM

#### N1. `compile/core.py:432` — `canonical_body` never extracted during merge
```python
duplicate_body = re.sub(r"^---\s*\n.*?\n---\s*\n", "", duplicate_content, flags=re.DOTALL)
merged_content = canonical_content.rstrip() + f"\n\n## Merged from {duplicate_name}\n\n{duplicate_body.strip()}\n"
```
`canonical_body` is never assigned. The merge appends `duplicate_body` to the full `canonical_content` (which still contains frontmatter). This works today because `_do_merge` only ever concatenates bodies into a canonical that already has correct frontmatter. However, if the merge logic ever needed to modify canonical frontmatter (e.g., merge tags, backlinks), this bug would surface. **Fix:** Add `canonical_body = re.sub(...)` and append only bodies.

### LOW

#### N2. Version drift across files

| File | Version |
|------|---------|
| `pyproject.toml` | `0.2.1` |
| `qmd_mcp.py` clientInfo | `0.3.0` |
| `docs/release/CHANGELOG.md` | `0.2.0` |
| `README.md` | no version stated |

Recommendation: single source of truth via `importlib.metadata.version("obsidian-llm-wiki")`.

#### N3. `qmd.py` hardcoded `"concepts"` collection

`cfg.qmd_collection` exists in `config.py` but is ignored at 5 call sites in `qmd.py`. Non-default collections are silently overridden.

#### N4. `extractors/web.py:300` — SPA wait strategy insufficient

```python
await page.goto(url, wait_until="domcontentloaded", timeout=timeout * 1000)
await asyncio.sleep(3)
```

`domcontentloaded` fires before JS-populated content renders. For SPAs (Medium, Substack), the 3s sleep may be insufficient. **Fix:** Add `await page.wait_for_load_state("networkidle")` after the sleep.

#### N5. flake8 cosmetic debt (33 issues)

All E127/E128 continuation indent, E251 spacing, E302/E303 blank lines, E203/E306 formatting, W391 trailing blank line. Zero functional impact.

## Honest Opinions

1. **This is genuinely good code.** Not "good for a side project" — good code, period. The Karpathy-style pipeline design is clean, and making template mode the default shows the author understands their own bottlenecks.

2. **The audit trail is excellent practice.** CODE_REVIEW, AUDIT, PATCH_NOTES, CHANGELOG, RELEASE.md — most repos twice this size don't maintain this discipline.

3. **The biggest remaining risk is the 600-second compile agent subprocess.** If the Hermes agent is down, the pipeline's post-processing silently degrades. Replace this with a direct Ollama call and this project jumps to 8.5+.

4. **The `_do_merge` O(N²) file scan in compile/core.py is a hidden scalability ceiling.** At 500+ notes it'll be perceptibly slow. A wikilink index + batch replace would be O(N log N).

5. **Version metadata drift (0.2.1 vs 0.3.0)** is a "death by a thousand cuts" issue. Easy fix, but it signals that release hygiene isn't fully automated.

6. **The store.py `_LockedConnection` design is pragmatic.** Not bulletproof against all race conditions, but good enough for a single-user PKM pipeline. I'd accept it as-is.

7. **820 passing tests is a lot.** The test infrastructure is solid. But 0.72x ratio means there's room for more edge-case coverage, especially around dead-code paths.

8. **This project shows clear evidence of being battle-tested.** Clipping workflow, review-before-write, telemetry, health checks, DLQ, circuit breakers — these aren't academic features. They came from real failures.

9. **The README still describes Ollama-local embedding as primary, but QMD MCP is the actual default.** Documentation drift is real and worth fixing before the next release.

10. **If I were using this daily, I'd be comfortable.** The bugs remaining are edge cases or performance ceilings, not data-loss risks. That's the hallmark of a codebase that's been through multiple review rounds.
